<section class="section">
  <div class="container">
    <div class="section-header">
      <span class="section-num">// 02 — SETUP</span>
      <h2 class="section-title">Step-by-Step Build Guide</h2>
      <p class="section-desc">Follow every step in order. Each command is explained so you understand the why, not just the what.</p>
    </div>
 
    <div class="steps-grid">
 
      <!-- STEP 1 -->
      <div class="step-card">
        <div class="step-head">
          <div class="step-num">01</div>
          <div>
            <div class="step-title">Launch & Prepare AWS EC2</div>
            <div class="step-subtitle">Ubuntu 22.04 LTS — t2.medium minimum recommended</div>
          </div>
        </div>
        <div class="step-body">
          <p>Log in to AWS Console → EC2 → Launch Instance. Choose <strong style="color:var(--text)">Ubuntu 22.04 LTS</strong>. Open ports 22 (SSH), 8080 (Jenkins UI), 80 (HTTP) in your Security Group.</p>
          <div class="callout callout-blue">
            <span class="callout-icon">💡</span>
            <p><strong>Why t2.medium?</strong> Jenkins needs RAM. A t2.micro with 1GB will freeze during Docker builds. Use at least 2GB RAM (t2.small) in dev. In production teams, Jenkins typically gets 4GB+.</p>
          </div>
          <div class="code-block">
            <div class="code-header">
              <span class="code-lang">bash — connect to your EC2</span>
              <button class="code-copy" onclick="copyCode(this)">copy</button>
            </div>
            <pre><span class="cm"># Replace with your key and EC2 public IP</span>
chmod 400 your-key.pem
ssh -i your-key.pem ubuntu@&lt;your-ec2-public-ip&gt;
 
<span class="cm"># Once inside, update the system first — always</span>
sudo apt update && sudo apt upgrade -y</pre>
          </div>
        </div>
      </div>
 
      <!-- STEP 2 -->
      <div class="step-card">
        <div class="step-head">
          <div class="step-num">02</div>
          <div>
            <div class="step-title">Install Java (Jenkins Dependency)</div>
            <div class="step-subtitle">Jenkins runs on the JVM — Java must come first</div>
          </div>
        </div>
        <div class="step-body">
          <p>Jenkins is a Java application. Without Java, it simply won't start. We install OpenJDK 17 — the LTS version Jenkins officially recommends.</p>
          <div class="code-block">
            <div class="code-header">
              <span class="code-lang">bash — install java 17</span>
              <button class="code-copy" onclick="copyCode(this)">copy</button>
            </div>
            <pre>sudo apt install openjdk-17-jdk -y
 
<span class="cm"># Verify installation</span>
java -version
<span class="cm"># Expected output:</span>
<span class="cm"># openjdk version "17.x.x" ...</span></pre>
          </div>
          <div class="callout callout-green">
            <span class="callout-icon">✅</span>
            <p><strong>Verify before moving on:</strong> Always run <code style="color:var(--green)">java -version</code> and confirm you see version 17. If you see version 11 or nothing, stop and fix it before installing Jenkins.</p>
          </div>
        </div>
      </div>
 
      <!-- STEP 3 -->
      <div class="step-card">
        <div class="step-head">
          <div class="step-num">03</div>
          <div>
            <div class="step-title">Install Jenkins</div>
            <div class="step-subtitle">The core CI/CD orchestration engine</div>
          </div>
        </div>
        <div class="step-body">
          <p>We install Jenkins from its official repository — not from Ubuntu's package manager — because Ubuntu's version is always outdated. Official repo gives us the latest LTS release.</p>
          <div class="code-block">
            <div class="code-header">
              <span class="code-lang">bash — add official jenkins repo & install</span>
              <button class="code-copy" onclick="copyCode(this)">copy</button>
            </div>
            <pre><span class="cm"># Add the Jenkins GPG key (security verification)</span>
curl -fsSL https://pkg.jenkins.io/debian-stable/jenkins.io-2023.key \
  | sudo tee /usr/share/keyrings/jenkins-keyring.asc > /dev/null
 
<span class="cm"># Add the Jenkins repository to apt sources</span>
echo deb [signed-by=/usr/share/keyrings/jenkins-keyring.asc] \
  https://pkg.jenkins.io/debian-stable binary/ \
  | sudo tee /etc/apt/sources.list.d/jenkins.list > /dev/null
 
<span class="cm"># Update apt and install Jenkins</span>
sudo apt update
sudo apt install jenkins -y
 
<span class="cm"># Enable and start the Jenkins service</span>
sudo systemctl enable jenkins
sudo systemctl start jenkins
 
<span class="cm"># Confirm Jenkins is running</span>
sudo systemctl status jenkins</pre>
          </div>
          <div class="callout">
            <span class="callout-icon">🔑</span>
            <p><strong>Get the initial admin password:</strong> Visit <code style="color:var(--accent2)">http://&lt;your-ec2-ip&gt;:8080</code> in your browser. Jenkins will ask for a key stored at <code style="color:var(--accent2)">/var/lib/jenkins/secrets/initialAdminPassword</code>. Run <code style="color:var(--accent2)">sudo cat /var/lib/jenkins/secrets/initialAdminPassword</code> to get it.</p>
          </div>
        </div>
      </div>
 
      <!-- STEP 4 -->
      <div class="step-card">
        <div class="step-head">
          <div class="step-num">04</div>
          <div>
            <div class="step-title">Install Docker</div>
            <div class="step-subtitle">The containerization engine for your builds</div>
          </div>
        </div>
        <div class="step-body">
          <p>Docker is what builds and runs your application containers. Again — we use the official Docker repo, not Ubuntu's outdated version.</p>
          <div class="code-block">
            <div class="code-header">
              <span class="code-lang">bash — install docker CE (community edition)</span>
              <button class="code-copy" onclick="copyCode(this)">copy</button>
            </div>
            <pre><span class="cm"># Install dependencies needed to add Docker's repo</span>
sudo apt install ca-certificates curl gnupg lsb-release -y
 
<span class="cm"># Add Docker's official GPG key</span>
sudo mkdir -p /etc/apt/keyrings
curl -fsSL https://download.docker.com/linux/ubuntu/gpg \
  | sudo gpg --dearmor -o /etc/apt/keyrings/docker.gpg
 
<span class="cm"># Add Docker repository</span>
echo "deb [arch=$(dpkg --print-architecture) \
  signed-by=/etc/apt/keyrings/docker.gpg] \
  https://download.docker.com/linux/ubuntu \
  $(lsb_release -cs) stable" \
  | sudo tee /etc/apt/sources.list.d/docker.list > /dev/null
 
<span class="cm"># Install Docker</span>
sudo apt update
sudo apt install docker-ce docker-ce-cli containerd.io -y
 
<span class="cm"># Start and enable Docker</span>
sudo systemctl enable docker
sudo systemctl start docker
 
<span class="cm"># Verify</span>
docker --version</pre>
          </div>
        </div>
      </div>
 
      <!-- STEP 5 — CRITICAL -->
      <div class="step-card" style="border-color: rgba(249,115,22,0.3);">
        <div class="step-head" style="background: rgba(249,115,22,0.05);">
          <div class="step-num">05</div>
          <div>
            <div class="step-title">⚠️ Fix Docker Permissions for Jenkins</div>
            <div class="step-subtitle">The #1 error DevOps beginners hit — do not skip this</div>
          </div>
        </div>
        <div class="step-body">
          <p>This is the most critical configuration step. Jenkins runs as its own system user (<code style="color:var(--blue)">jenkins</code>). By default, only the <code style="color:var(--blue)">root</code> user and <code style="color:var(--blue)">docker</code> group can run Docker commands. If you don't add Jenkins to the docker group, every single pipeline build will fail.</p>
          <div class="code-block">
            <div class="code-header">
              <span class="code-lang">bash — grant jenkins docker access</span>
              <button class="code-copy" onclick="copyCode(this)">copy</button>
            </div>
            <pre><span class="cm"># Add the 'jenkins' user to the 'docker' group</span>
sudo usermod -aG docker jenkins
 
<span class="cm"># Restart Jenkins to apply group membership</span>
sudo systemctl restart jenkins
 
<span class="cm"># Verify docker group includes jenkins</span>
grep docker /etc/group
<span class="cm"># You should see: docker:x:999:jenkins</span></pre>
          </div>
          <div class="callout">
            <span class="callout-icon">🧠</span>
            <p><strong>Why does this work?</strong> The Docker daemon (dockerd) listens on a Unix socket at <code style="color:var(--accent2)">/var/run/docker.sock</code>. That socket is owned by the <code style="color:var(--accent2)">docker</code> group. Adding Jenkins to this group gives it read/write access to the socket — which means it can issue docker commands as if it were any other user in that group.</p>
          </div>
        </div>
      </div>
 
      <!-- STEP 6 -->
      <div class="step-card">
        <div class="step-head">
          <div class="step-num">06</div>
          <div>
            <div class="step-title">Write the Jenkinsfile</div>
            <div class="step-subtitle">Pipeline as Code — the DevOps standard</div>
          </div>
        </div>
        <div class="step-body">
          <p>The <code style="color:var(--blue)">Jenkinsfile</code> lives in the root of your repository. It tells Jenkins exactly what to do — in order — every time a build runs. This is called <strong style="color:var(--text)">Pipeline as Code</strong>, and it's how every serious DevOps team works.</p>
          <div class="code-block">
            <div class="code-header">
              <span class="code-lang">groovy — Jenkinsfile (declarative syntax)</span>
              <button class="code-copy" onclick="copyCode(this)">copy</button>
            </div>
            <pre><span class="fn">pipeline</span> {
  <span class="kw">agent</span> any  <span class="cm">// run on any available Jenkins agent</span>
 
  <span class="kw">environment</span> {
    <span class="var">IMAGE_NAME</span> = <span class="str">'notes-app'</span>
    <span class="var">IMAGE_TAG</span>  = <span class="str">'latest'</span>
  }
 
  <span class="kw">stages</span> {
 
    <span class="fn">stage</span>(<span class="str">'Checkout'</span>) {
      <span class="kw">steps</span> {
        <span class="cm">// Pull the latest code from GitHub</span>
        git url: <span class="str">'https://github.com/arkantandel/notes-app.git'</span>,
            branch: <span class="str">'main'</span>
      }
    }
 
    <span class="fn">stage</span>(<span class="str">'Build Docker Image'</span>) {
      <span class="kw">steps</span> {
        <span class="cm">// Build the container image from the Dockerfile</span>
        sh <span class="str">'docker build -t ${IMAGE_NAME}:${IMAGE_TAG} .'</span>
      }
    }
 
    <span class="fn">stage</span>(<span class="str">'Test'</span>) {
      <span class="kw">steps</span> {
        <span class="cm">// Run your test suite inside a container</span>
        sh <span class="str">'docker run --rm ${IMAGE_NAME}:${IMAGE_TAG} npm test'</span>
      }
    }
 
    <span class="fn">stage</span>(<span class="str">'Deploy'</span>) {
      <span class="kw">steps</span> {
        <span class="cm">// Stop old container, run new one</span>
        sh <span class="str">'''
          docker stop notes-app-container || true
          docker rm   notes-app-container || true
          docker run -d -p 80:3000 --name notes-app-container \
            ${IMAGE_NAME}:${IMAGE_TAG}
        '''</span>
      }
    }
 
  }
 
  <span class="kw">post</span> {
    <span class="kw">success</span> { echo <span class="str">'Pipeline succeeded — app is live!'</span> }
    <span class="kw">failure</span> { echo <span class="str">'Pipeline failed — check the logs above'</span> }
  }
}</pre>
          </div>
          <div class="callout callout-blue">
            <span class="callout-icon">📘</span>
            <p><strong>Why Declarative over Scripted?</strong> Declarative syntax (what we used above) is structured, readable, and easier to lint and validate. Scripted syntax is more powerful but harder to maintain. For 95% of real pipelines, Declarative is the right choice.</p>
          </div>
        </div>
      </div>

      <span class="section-num">// 03 — DEBUGGING</span>
      <h2 class="section-title">Real Errors & How to Fix Them</h2>
      <p class="section-desc">These are actual errors from this build. Understanding errors is what separates engineers from copy-pasters.</p>
    </div>
    <div class="error-grid">
 
      <div class="error-card">
        <div class="error-head">
          <span class="error-badge">ERROR 01</span>
          <span class="error-title">Docker Permission Denied</span>
        </div>
        <div class="error-body">
          <div class="error-msg">Got permission denied while trying to connect to the Docker daemon socket at unix:///var/run/docker.sock</div>
          <span class="fix-label">▶ ROOT CAUSE</span>
          <p class="fix-explain" style="margin-bottom:12px;">Jenkins runs as user <code style="color:var(--blue);font-size:12px;">jenkins</code>, which is not in the <code style="color:var(--blue);font-size:12px;">docker</code> group. The Docker socket only accepts connections from root or members of that group.</p>
          <span class="fix-label">▶ FIX</span>
          <div class="code-block"><div class="code-header"><span class="code-lang">bash</span></div><pre>sudo usermod -aG docker jenkins && sudo systemctl restart jenkins</pre></div>
        </div>
      </div>
 
      <div class="error-card">
        <div class="error-head">
          <span class="error-badge">ERROR 02</span>
          <span class="error-title">Docker Build Context Error</span>
        </div>
        <div class="error-body">
          <div class="error-msg">unable to evaluate symlinks in Dockerfile path: lstat /var/lib/jenkins/workspace/.../Dockerfile: no such file or directory</div>
          <span class="fix-label">▶ ROOT CAUSE</span>
          <p class="fix-explain" style="margin-bottom:12px;">Jenkins checked out the code into a subdirectory, but the <code style="color:var(--blue);font-size:12px;">docker build</code> command was running from the wrong working directory — one level above where the Dockerfile actually lived.</p>
          <span class="fix-label">▶ FIX</span>
          <div class="code-block"><div class="code-header"><span class="code-lang">groovy — in Jenkinsfile</span></div><pre><span class="cm">// Add dir() block to set correct working directory</span>
dir(<span class="str">'notes-app'</span>) {
  sh <span class="str">'docker build -t notes-app:latest .'</span>
}</pre></div>
        </div>
      </div>
 
      <div class="error-card">
        <div class="error-head">
          <span class="error-badge">ERROR 03</span>
          <span class="error-title">Docker Buildx Plugin Crash</span>
        </div>
        <div class="error-body">
          <div class="error-msg">ERROR: failed to solve: failed to read dockerfile: open .../Dockerfile: no such file or directory — buildx plugin failed</div>
          <span class="fix-label">▶ ROOT CAUSE</span>
          <p class="fix-explain" style="margin-bottom:12px;">The Docker Buildx plugin (used for multi-architecture builds) was corrupted or had a stale state. This can happen after a system update or incomplete installation.</p>
          <span class="fix-label">▶ FIX</span>
          <div class="code-block"><div class="code-header"><span class="code-lang">bash — remove and reinstall buildx</span></div><pre><span class="cm"># Remove corrupted buildx plugin</span>
rm -rf ~/.docker/cli-plugins/docker-buildx
rm -rf /usr/lib/docker/cli-plugins/docker-buildx
 
<span class="cm"># Reinstall via apt</span>
sudo apt install --reinstall docker-buildx-plugin -y
 
<span class="cm"># Verify</span>
docker buildx version</pre></div>
        </div>
      </div>

<script>
function copyCode(btn) {
  const pre = btn.closest('.code-block').querySelector('pre');
  const text = pre.innerText;
  navigator.clipboard.writeText(text).then(() => {
    btn.textContent = 'copied!';
    btn.style.color = '#3fb950';
    btn.style.borderColor = '#3fb950';
    setTimeout(() => { btn.textContent = 'copy'; btn.style.color = ''; btn.style.borderColor = ''; }, 2000);
  });
}
</script>
